1. Closures

A closure is a function that "remembers" variables from its outer scope, even after the outer function has returned.

Why care? Closures are how JavaScript does data privacy, callbacks, and event handlers. You've already used them without knowing it!
// A counter using closures function createCounter(startValue) { let count = startValue; // ← this variable is "closed over" return { increment() { return ++count; }, decrement() { return --count; }, getCount() { return count; } }; // After createCounter() returns, `count` should be garbage collected... // But it ISN'T! The returned functions still reference it. // That's a closure. } const counter = createCounter(0); console.log(counter.increment()); // 1 console.log(counter.increment()); // 2 console.log(counter.decrement()); // 1 // Can you access `count` directly? NO! It's private. // console.log(count); ← ReferenceError
Break It: What happens if you call createCounter() twice? Do the two counters share state or have independent state? Click "Run" to find out!

2. Promises & Async/Await

Promises represent a value that might not exist YET but will in the future (or will fail).

Analogy: A promise is like ordering food at a restaurant. You get a receipt (the promise). The food isn't ready yet, but the receipt guarantees you'll either get food (.then) or an apology (.catch).
// Simulating a network request with setTimeout function fetchBook(id) { return new Promise((resolve, reject) => { console.log(`Fetching book ${id}...`); setTimeout(() => { if (id > 0) { resolve({ id, title: "Async Book", author: "Promise" }); } else { reject(new Error("Invalid ID!")); } }, 1000); // Simulates 1 second network delay }); } // Using .then/.catch (old way): fetchBook(1) .then(book => console.log("Got:", book.title)) .catch(err => console.error("Error:", err.message)); // Using async/await (modern way — same thing, cleaner syntax): async function getBook() { try { const book = await fetchBook(1); console.log("Got:", book.title); } catch (err) { console.error("Error:", err.message); } }

3. The Event Loop

JavaScript is single-threaded but non-blocking. The event loop is HOW.

Mental model:
1. Call Stack — currently executing code (one thing at a time)
2. Web APIs — browser handles timers, network, DOM events in background
3. Callback Queue — completed callbacks waiting to run
4. Microtask Queue — Promises (.then callbacks) — higher priority than callback queue!
5. Event Loop — moves tasks from queues → call stack when stack is empty
// PREDICT the order before running! console.log("1: Start"); setTimeout(() => { console.log("2: setTimeout callback"); }, 0); // 0ms delay — but still goes to callback queue! Promise.resolve().then(() => { console.log("3: Promise.then (microtask)"); }); console.log("4: End"); // Expected output: 1, 4, 3, 2 // WHY? // "Start" and "End" run immediately (call stack) // Promise.then → microtask queue (higher priority) // setTimeout → callback queue (lower priority) // Event loop: drain microtasks FIRST, then callbacks
Key Insight: setTimeout(fn, 0) does NOT mean "run immediately." It means "run as soon as the call stack AND microtask queue are empty." That's why Promise.then runs before setTimeout even with 0ms delay!

4. Scope & Hoisting

Understanding where variables "live" and when they become available.

// var is HOISTED (declaration moves to top, value stays) console.log(x); // undefined (not error! var is hoisted) var x = 5; // let/const are NOT hoisted (well, they are, but in a "temporal dead zone") // console.log(y); ← ReferenceError: Cannot access 'y' before initialization let y = 10; // Scope difference: function scopeDemo() { if (true) { var varScoped = "I escape the if block!"; let letScoped = "I'm trapped in the if block."; } console.log(varScoped); // Works! var is function-scoped // console.log(letScoped); ← ReferenceError: let is block-scoped } // Classic interview question: the loop problem for (var i = 0; i < 3; i++) { setTimeout(() => console.log("var i:", i), 100); } // Prints: 3, 3, 3 (NOT 0, 1, 2!) — because var is shared across iterations for (let j = 0; j < 3; j++) { setTimeout(() => console.log("let j:", j), 100); } // Prints: 0, 1, 2 — because let creates a NEW variable per iteration

5. Array Methods (map, filter, reduce)

Functional programming basics — transform data without loops.

const books = [ { title: "Clean Code", year: 2008, pages: 431 }, { title: "SICP", year: 1996, pages: 657 }, { title: "You Don't Know JS", year: 2014, pages: 278 }, { title: "Design Patterns", year: 1994, pages: 395 }, ]; // FILTER: keep only items matching a condition const modern = books.filter(b => b.year >= 2000); // → [Clean Code, You Don't Know JS] // MAP: transform each item into something else const titles = books.map(b => b.title); // → ["Clean Code", "SICP", "You Don't Know JS", "Design Patterns"] // REDUCE: collapse array into a single value const totalPages = books.reduce((sum, b) => sum + b.pages, 0); // → 1761 // CHAIN them! Get total pages of books after year 2000: const modernPages = books .filter(b => b.year >= 2000) .reduce((sum, b) => sum + b.pages, 0); // → 709
